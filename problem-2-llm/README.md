# 🤖 LLM Evaluation Framework

A modular, offline Large Language Model (LLM) evaluation framework built using **Python**, **Streamlit**, and **Ollama**. The framework evaluates responses generated by LLMs using an independent judge model, measures evaluation quality, detects common evaluation biases, validates judge reliability, and generates comprehensive reports.

---

# Project Overview

Evaluating Large Language Models is becoming increasingly important as they are used in education, software development, healthcare, customer support, and decision-making systems. Traditional evaluation methods often rely on human annotators, which are expensive, time-consuming, and difficult to scale.

Recent research demonstrates that LLMs themselves can be used as automated evaluators. However, using one LLM to judge another introduces several challenges, including:

- Position Bias
- Verbosity Bias
- Self-Enhancement Bias
- Sycophancy Bias
- Score Clustering
- Inconsistent Judgements

This project implements a complete LLM evaluation framework that addresses these challenges while remaining fully offline using Ollama.

The framework generates responses using one language model, evaluates them using an independent judge model, computes multiple evaluation metrics, detects bias, validates judge consistency, and produces downloadable reports through an interactive Streamlit dashboard.

---

# Problem Statement

The objective of this project is to design and implement a robust evaluation framework for Large Language Models that satisfies the following requirements:

- Load test suites from structured files.
- Generate responses using a configurable LLM.
- Evaluate responses using an independent judge model.
- Score responses across multiple evaluation criteria.
- Detect common LLM evaluation biases.
- Validate judge consistency.
- Generate machine-readable evaluation reports.
- Provide an interactive user interface for running evaluations.

Unlike traditional benchmarking systems that focus only on model accuracy, this framework also measures the reliability and fairness of the evaluation process itself.

This separation of concerns improves maintainability, readability, extensibility, and testing. Each stage of the evaluation pipeline has a dedicated responsibility, making it possible to modify or replace one component without affecting the rest of the system.

# Architecture Overview

The framework processes every evaluation through a sequence of independent modules. Each module performs one specific task and passes structured data to the next module.

The complete workflow is:

1. Load the evaluation dataset.
2. Generate prompts.
3. Produce responses from the generator model.
4. Evaluate responses using an independent judge.
5. Parse the judge's structured output.
6. Calculate evaluation metrics.
7. Detect evaluation biases.
8. Validate judge consistency.
9. Generate reports.
10. Display results in the Streamlit dashboard.

This layered design keeps the evaluation process transparent and easy to debug.

# Architectural Principles

The framework was designed around several software engineering principles.

## 1. Separation of Concerns

Each service has a single responsibility.

For example:

- The loader only loads test suites.
- The generator only generates responses.
- The judge only evaluates responses.
- The metrics module only computes statistics.
- The reporter only exports reports.

Because each module performs one task, the codebase is easier to understand and maintain.

---

## 2. Modularity

Every major feature exists as an independent service.

Examples include:

- Loader
- Generator
- Judge
- Parser
- Metrics
- Bias Manager
- Validator
- Reporter

New functionality can be added without modifying the existing modules.

For example, adding another bias detector only requires creating a new detector class and registering it with the Bias Manager.

---

## 3. Reusability

Most services are reusable across different projects.

Examples include:

- PromptBuilder
- MetricsCalculator
- ReportGenerator
- JudgeValidator

These components are not tightly coupled to Streamlit and can be reused in command-line tools, APIs, or other applications.

The framework uses **Pydantic** models throughout the application.

Benefits include:

- Automatic validation
- Strong typing
- Easier debugging
- Safer JSON parsing
- Improved code readability

Every major object is represented using a schema rather than unstructured dictionaries.

Examples include:

- TestCase
- JudgeCall
- JudgeVerdict
- EvaluationResult
- CriterionScore

---

## 5. Extensibility

The architecture was designed to support future expansion.

Possible future improvements include:

- Additional LLM providers
- Multiple judge models
- Additional bias detectors
- Database storage
- REST API
- Multi-user dashboard

These features can be added without redesigning the existing architecture.

# Evaluation Pipeline

The Evaluation Pipeline is the central component of the framework. It orchestrates the complete evaluation process by coordinating all services, from loading a test case to producing the final evaluation report.

Rather than embedding all evaluation logic in a single module, the framework delegates each responsibility to a dedicated service. This modular approach improves readability, testing, and future extensibility.

The pipeline follows the same sequence for every test case, ensuring that all evaluations are reproducible and consistent.

---

# Pipeline Overview

The complete evaluation process is illustrated below.

```text
                 Test Suite
                      │
                      ▼
             TestSuiteLoader
                      │
                      ▼
             Prompt Builder
                      │
         ┌────────────┴────────────┐
         │                         │
         ▼                         ▼
  Generator Model A        Generator Model B
         │                         │
         └────────────┬────────────┘
                      ▼
             Independent Judge
                      │
                      ▼
              Verdict Parser
                      │
                      ▼
             Evaluation Result
                      │
         ┌────────────┼──────────────┐
         ▼            ▼              ▼
     Metrics      Bias Tests    Validation
         │            │              │
         └────────────┴──────────────┘
                      ▼
              Report Generator
                      │
                      ▼
          JSON / CSV / Dashboard
```

Every stage in this pipeline performs one specific task before passing structured data to the next stage.

---

# Step 1 – Load Test Suite

The evaluation begins by loading a structured test suite from a JSON file.

Each test case contains:

- Unique identifier
- Input question
- System prompt
- Expected output

Example:

```json
{
    "id": 1,
    "input": "What is Python?",
    "system_prompt": "Answer in 2-3 sentences.",
    "expected_output": "Python is a high-level programming language..."
}
```

The loader validates every test case using Pydantic before it enters the evaluation pipeline.

This ensures that malformed datasets are detected early and do not interrupt later stages.

---

# Step 2 – Prompt Construction

Once the test case is loaded, prompts are constructed for the generator model.

The prompt builder combines:

- User question
- System instruction

Example:

```
Question:
What is Python?

Instructions:
Answer in 2–3 sentences.
```

Separating prompt construction from generation makes prompt engineering easier because prompts can be modified without changing the generator implementation.

---

# Step 3 – Response Generation

The framework generates two independent responses.

```
Answer A
```

and

```
Answer B
```

Each answer is generated using the configured generator model.

The generator service is responsible for:

- Sending prompts to Ollama
- Receiving model responses
- Returning structured text

Separating generation into its own service allows the framework to support different models in the future without modifying the evaluation pipeline.

---

# Why Two Responses?

The framework compares two candidate responses rather than evaluating a single response in isolation.

Comparative evaluation is generally easier for language models because they only need to determine which response is better instead of assigning an absolute quality score.

This approach also mirrors many recent LLM evaluation benchmarks where pairwise comparison is preferred over independent scoring.

---

# Step 4 – Independent Judge Evaluation

After both responses have been generated, they are passed to an independent judge model.

The judge receives:

- Original question
- Expected answer (if available)
- Response A
- Response B

The judge evaluates both responses according to a predefined rubric.

Current evaluation criteria include:

- Correctness
- Faithfulness
- Completeness
- Instruction Following
- Tone
- Safety

The judge returns a structured JSON object containing:

- Winner
- Overall scores
- Criterion-wise scores
- Detailed rationale

Using structured JSON simplifies downstream processing and minimizes manual parsing.

---

# Why an Independent Judge?

Using the same language model for both generation and evaluation may introduce bias.

For example, a model may systematically prefer responses written in its own style.

To reduce this risk, the framework uses:

- Generator → Llama 3.1
- Judge → Qwen 2.5

This separation reduces self-enhancement bias and produces more objective evaluations.

---

# Step 5 – Verdict Parsing

Although the judge is instructed to return JSON, LLM outputs may occasionally contain formatting inconsistencies.

The parser is responsible for:

- Extracting JSON
- Validating the schema
- Converting raw output into Pydantic models

This stage improves the robustness of the framework by preventing malformed responses from propagating through the system.

---

# Step 6 – Evaluation Result

After parsing, the framework creates a complete EvaluationResult object.

The object contains:

- Original test case
- Generated Answer A
- Generated Answer B
- Judge verdict
- Judge metadata

This object becomes the central data structure used throughout the remainder of the framework.

---

# Step 7 – Metric Calculation

Once all test cases have been evaluated, the Metrics Calculator computes aggregate statistics.

These include:

- Total test cases
- Pass rate
- Win rate
- Average scores
- Criterion-wise averages

These metrics provide a quantitative summary of model performance across the entire dataset.

---

# Step 8 – Bias Analysis

Evaluation quality depends not only on scores but also on fairness.

The framework therefore performs multiple bias detection experiments.

Implemented detectors include:

- Position Bias
- Verbosity Bias
- Self-Enhancement Bias
- Sycophancy Bias
- Score Clustering

Each detector analyzes the evaluation results independently and produces its own report.

This allows the framework to assess the reliability of the judge rather than only the quality of generated answers.

---

# Step 9 – Judge Validation

The framework validates judge reliability using Test-Retest Reliability.

The same evaluation is repeated multiple times.

Example:

```
Run 1 → Winner A

Run 2 → Winner A

Run 3 → Winner A
```

If the judge consistently produces the same verdict, it is considered reliable.

The framework computes an overall consistency percentage based on repeated evaluations.

---

# Step 10 – Report Generation

Finally, all evaluation outputs are combined into structured reports.

Generated reports include:

## JSON Report

Contains:

- Metadata
- Metrics
- Bias Analysis
- Judge Validation
- Recommendations
- Individual Evaluations

The JSON report is intended for programmatic use.

---

## CSV Report

Contains:

- Test Case ID
- Winner
- Scores
- Judge Model
- Timestamp
- Generated Responses

The CSV report is intended for spreadsheet analysis and manual inspection.

---

# Step 11 – Dashboard Visualization

The Streamlit dashboard serves as the user interface for the framework.

Users can:

- Upload a test suite
- Configure models
- Execute evaluations
- View metrics
- Inspect bias reports
- Validate judge reliability
- Explore individual results
- Download reports

The dashboard separates presentation from business logic by consuming outputs produced by the evaluation pipeline rather than performing computations itself.

# Test Suite Format

The quality of any evaluation framework depends heavily on the quality of the evaluation dataset. A well-designed test suite ensures that all models are evaluated consistently using the same inputs and expected outputs.

In this framework, test cases are stored as **JSON** documents. JSON was selected because it is lightweight, human-readable, language-independent, and natively supported by Python.

Each uploaded JSON file contains a list of independent test cases. Every test case represents one evaluation instance and contains all the information required to generate and evaluate model responses.

---

# Test Case Structure

Each test case follows the schema below.

```json
{
    "id": 1,
    "input": "What is Python?",
    "system_prompt": "Answer in 2-3 sentences.",
    "expected_output": "Python is a high-level programming language used for web development, AI, automation, and more."
}
```

---

# Test Case Fields

## id

The unique identifier for the test case.

Purpose:

- Uniquely identifies each evaluation.
- Simplifies report generation.
- Enables result tracking.

Example:

```json
"id": 5
```

---

## input

The question or task presented to the generator model.

Examples include:

- Knowledge questions
- Programming questions
- Reasoning tasks
- Summarization
- Translation
- Classification

Example:

```json
"input": "Explain REST API."
```

---

## system_prompt

Defines the instruction given to the generator model.

Examples:

```
Answer in one sentence.
```

```
Explain using bullet points.
```

```
Use simple language.
```

```
Keep the answer under 100 words.
```

Separating the system prompt from the question makes the framework more flexible because different prompting strategies can be evaluated without modifying the question itself.

---

## expected_output

Represents the ideal or reference answer.

The expected output serves as additional context for the judge model during evaluation.

It is **not** copied or compared directly.

Instead, it provides a reference that helps the judge determine:

- Correctness
- Completeness
- Faithfulness

If an expected answer is unavailable, the framework still performs evaluation using only the question and generated responses.

# Prompt Engineering

Prompt engineering plays a critical role in LLM evaluation.

Even small changes to prompts can significantly affect generated responses and judge decisions.

To improve reproducibility, all prompts are constructed automatically by the framework instead of being manually assembled during evaluation.

This ensures that every evaluation follows the same prompt template.

---

# Generator Prompt

The generator prompt is responsible for producing candidate responses.

The framework constructs the prompt using:

- Question
- System instruction

Example:

```
Question:
What is Python?

Instructions:
Answer in 2–3 sentences.
```

The generator only receives the information necessary to answer the question.

It does **not** receive:

- Expected answer
- Judge rubric
- Evaluation criteria

This prevents leakage of evaluation information into the generated response.

---

# Why Keep Generator Prompts Simple?

The generator prompt intentionally contains only the information required to answer the question.

Advantages:

- Reduces prompt bias
- Encourages natural responses
- Simplifies experimentation
- Improves reproducibility

A concise prompt also makes it easier to compare different generator models because each model receives identical instructions.

---

# Judge Prompt

The judge prompt differs significantly from the generator prompt.

Instead of asking the model to answer a question, it asks the model to compare two candidate responses.

The judge receives:

- Original question
- Expected answer (if available)
- Answer A
- Answer B
- Evaluation rubric

The rubric includes:

- Correctness
- Faithfulness
- Completeness
- Instruction Following
- Tone
- Safety

The judge is instructed to return only valid JSON.

This structured output simplifies parsing and reduces ambiguity.

---

# Why Separate Generator and Judge Prompts?

Generation and evaluation are fundamentally different tasks.

The generator prompt focuses on producing a high-quality response.

The judge prompt focuses on comparing responses objectively using predefined evaluation criteria.

Separating these prompts provides several benefits:

- Cleaner architecture
- Better prompt maintainability
- Independent optimization
- Reduced bias

Future improvements to one prompt do not affect the other.

---

# Prompt Consistency

One challenge in LLM evaluation is ensuring that every model receives the same instructions.

To address this, prompts are generated programmatically rather than manually.

This guarantees:

- Identical formatting
- Consistent wording
- Reproducible evaluations

Prompt consistency is essential for fair benchmarking because differences in prompt wording may influence model performance.

---

# Structured Judge Output

The judge is instructed to produce structured JSON rather than free-form text.

Example:

```json
{
    "winner": "A",
    "overall_score_a": 8,
    "overall_score_b": 6,
    "criteria": [
        {
            "name": "Correctness",
            "score_a": 9,
            "score_b": 7
        }
    ]
}
```

Structured output provides several advantages:

- Easier parsing
- Reliable validation
- Machine-readable reports
- Automatic metric calculation

The parser validates this JSON using Pydantic before passing it to the remainder of the evaluation pipeline.

---

The prompt design follows several principles.

## Simplicity

Prompts contain only the information necessary for the task.

---

## Consistency

Every evaluation uses the same prompt structure.

---

## Separation of Responsibilities

Generator prompts create responses.

Judge prompts evaluate responses.

---

## Structured Outputs

The judge always returns machine-readable JSON.

---

## Reproducibility

Automatically generated prompts ensure that repeated evaluations are consistent across runs and models.

The framework separates evaluation data from prompt construction, making the system more flexible and easier to maintain.

By storing test cases as structured JSON documents and generating prompts programmatically, the framework ensures consistent, reproducible, and fair evaluations across different language models.

# Evaluation Criteria and Metrics

The primary goal of this framework is not only to generate responses from Large Language Models but also to evaluate them in a fair, consistent, and reproducible manner.

Rather than assigning a single quality score, the framework evaluates responses across multiple dimensions. This provides a more comprehensive assessment of response quality and reduces the likelihood that one strong or weak characteristic dominates the overall evaluation.

The evaluation process is divided into two parts:

1. **Qualitative Evaluation Criteria** – Individual aspects of response quality.
2. **Quantitative Evaluation Metrics** – Numerical statistics calculated from all evaluation results.

---

# Evaluation Criteria

The judge evaluates every pair of responses using six predefined criteria.

These criteria were selected because they cover both the factual quality of the response and its usefulness to the user.

---

## 1. Correctness

Correctness measures whether the generated response is factually accurate.

The judge determines whether:

- The information is correct.
- Facts are accurate.
- Statements are logically valid.

Example:

Question:

```
What is Python?
```

Correct Answer:

```
Python is a programming language.
```

Incorrect Answer:

```
Python is a snake species only.
```

A response with factual errors receives a lower correctness score.

---

## 2. Faithfulness

Faithfulness measures how closely the response answers the original question without introducing misleading or unrelated information.

The judge evaluates:

- Relevance to the question.
- Accuracy relative to the expected answer.
- Absence of hallucinated information.

Example:

Question:

```
Explain REST API.
```

A faithful response explains REST APIs.

A response discussing GraphQL instead would receive a lower faithfulness score.

---

## 3. Completeness

Completeness measures whether the response covers all important aspects of the question.

The judge considers:

- Missing information.
- Partial explanations.
- Overall coverage.

Example:

Question:

```
What is Docker?
```

A complete answer should explain:

- Containers
- Portability
- Application packaging

An answer mentioning only containers would receive a lower completeness score.

---

## 4. Instruction Following

Instruction Following evaluates whether the response obeys the supplied system prompt.

Examples:

Instruction:

```
Answer in one sentence.
```

A two-paragraph response receives a lower score.

Instruction:

```
Use bullet points.
```

A paragraph response may lose points.

This criterion measures compliance rather than factual quality.

---

## 5. Tone

Tone evaluates the writing style and appropriateness of the response.

The judge considers:

- Professionalism
- Clarity
- Readability
- Neutrality

Responses that are offensive, confusing, or poorly written receive lower scores.

---

## 6. Safety

Safety evaluates whether the response contains harmful, dangerous, or inappropriate content.

The judge checks for:

- Harmful advice
- Toxic language
- Unsafe recommendations
- Malicious instructions

Safe responses receive higher scores regardless of their factual content.

---

# Scoring Methodology

Each criterion is scored independently for:

- Answer A
- Answer B

Scores range from:

```
0 – 10
```

Where:

| Score | Interpretation |
|--------|---------------|
| 0–2 | Very Poor |
| 3–4 | Poor |
| 5–6 | Average |
| 7–8 | Good |
| 9–10 | Excellent |

The judge also selects a winner for each criterion before determining the overall winner.

---

# Overall Winner

After evaluating all criteria, the judge determines the better response.

Possible outcomes:

```
Winner = A
```

```
Winner = B
```

```
Winner = Tie
```

The overall winner is based on the combined evaluation rather than a single criterion.

---

# Bias Detection

Evaluating Large Language Models involves more than measuring response quality. An evaluation system must also ensure that the **judge itself** behaves fairly and consistently.

Recent research has shown that LLM-based judges are susceptible to several systematic biases that can influence evaluation outcomes. These biases may cause the judge to prefer responses for reasons unrelated to their actual quality.

To address this challenge, this framework implements five independent bias detectors.

The implemented detectors are:

- Position Bias
- Verbosity Bias
- Self-Enhancement Bias
- Sycophancy Bias
- Score Clustering

Each detector analyzes evaluation results independently and produces quantitative measurements that help assess the reliability of the judge.

---

# Why Bias Detection?

Consider the following scenario.

Two responses contain identical information.

```
Answer A

Python is a programming language.
```

```
Answer B

Python is a programming language.
```

If the judge consistently prefers Answer A simply because it appears first, the evaluation is unfair.

Similarly, if the judge always prefers longer responses regardless of quality, then response scores become misleading.

Therefore, detecting bias is essential for trustworthy automated evaluation.

---

# 1. Position Bias

## Definition

Position Bias occurs when the judge consistently favors one answer because of its position rather than its quality.

For example,

```
Judge

Answer A

Answer B

Winner

A
```

Now reverse the order.

```
Judge

Answer B

Answer A

Winner

B
```

If the winner changes only because the answers were swapped, position bias exists.

---

## Detection Method

For every evaluation, the framework performs two comparisons.

Original order

```
Judge(A, B)
```

Reversed order

```
Judge(B, A)
```

The winners are then compared.

---

## Flip Rate

The framework computes the Flip Rate.

Formula

```
               Flipped Cases
Flip Rate = -------------------- ×100
              Total Test Cases
```

Where

Flipped Cases represent evaluations whose winner changed after reversing the answer order.

---

## Interpretation

| Flip Rate | Interpretation |
|------------|---------------|
| 0% | No position bias detected |
| Low | Minor position preference |
| High | Significant position bias |

---

## Implementation

The Position Bias Detector:

1. Uses the original evaluation result.
2. Re-evaluates only the reversed answer order.
3. Compares both winners.
4. Computes the Flip Rate.

This approach minimizes unnecessary judge calls while still detecting positional preferences.

---

# 2. Verbosity Bias

## Definition

Verbosity Bias occurs when the judge consistently prefers longer responses even when they are not objectively better.

Example

Answer A

```
Python is a programming language.
```

Answer B

```
Python is a high-level interpreted object-oriented programming language developed...
```

Although Answer B is much longer, it is not necessarily better.

---

## Detection Method

For every evaluation, the framework computes:

- Length of Answer A
- Length of Answer B
- Judge Winner

The detector then checks whether the judge consistently prefers the longer response.

---

## Verbosity Bias Rate

Formula

```
                Longer Answer Wins
Bias Rate = --------------------------- ×100
               Total Test Cases
```

---

## Additional Statistics

The detector also records

- Answer Length
- Length Difference
- Length Ratio

These statistics help determine whether response length influenced the final decision.

---

## Interpretation

| Bias Rate | Interpretation |
|-----------|---------------|
| 0% | No verbosity bias |
| Low | Slight preference for longer answers |
| High | Strong verbosity bias |

---

# 3. Self-Enhancement Bias

## Definition

Self-Enhancement Bias occurs when a judge favors responses generated by its own model family.

For example,

Generator

```
Llama
```

Judge

```
Llama
```

The judge may unconsciously prefer responses similar to its own generation style.

---

## Mitigation Strategy

Rather than detecting this bias from scores alone, the framework reduces the risk by design.

Generator

```
Llama 3.1
```

Judge

```
Qwen 2.5
```

Since the models belong to different families, self-preference is less likely.

---

## Detection

The detector compares

Generator Family

vs

Judge Family

If both belong to the same family, the evaluation is marked as higher risk.

---

## Risk Levels

| Risk | Meaning |
|------|---------|
| Low | Different model families |
| Medium | Similar architecture |
| High | Same model family |

---

## Why This Matters

Using independent model families improves evaluation fairness and follows recommendations from recent LLM evaluation research.

---

# 4. Sycophancy Bias

## Definition

Sycophancy Bias occurs when a judge prefers responses that sound confident or authoritative even if they are incorrect.

Example

Answer A

```
Python is a programming language.
```

Answer B

```
Without any doubt, Python is definitely a snake.
```

Although Answer B sounds confident, it is factually incorrect.

A reliable judge should still prefer Answer A.

---

## Detection Method

The detector identifies intentionally overconfident but incorrect responses.

It then checks whether the judge selected those responses.

---

## Sycophancy Rate

Formula

```
               Fooled Cases
Rate = ------------------------- ×100
             Total Test Cases
```

---

## Interpretation

| Rate | Interpretation |
|------|---------------|
| 0% | Judge resisted sycophancy |
| High | Judge influenced by confidence |

---

# 5. Score Clustering

## Definition

Score Clustering occurs when the judge repeatedly assigns nearly identical scores regardless of response quality.

Example

```
8

8

8

8

8

8

8
```

Such scoring provides poor discrimination between responses.

---

## Detection Method

The framework analyzes all overall scores.

Computed statistics include:

- Mean
- Median
- Minimum
- Maximum
- Variance
- Standard Deviation

---

## Standard Deviation

Formula

```
        Σ(x−μ)²
σ = √ -------------
            N
```

A very small standard deviation indicates excessive score clustering.

---

## Interpretation

| Standard Deviation | Interpretation |
|--------------------|---------------|
| Very Low | Possible score clustering |
| Moderate | Healthy score distribution |
| High | Diverse scoring |

---

## Why Standard Deviation?

Standard deviation measures how spread out the assigned scores are.

A judge that always assigns similar scores provides less useful evaluations than one capable of distinguishing between responses of different quality.

---

The framework combines all five detectors to provide an overall assessment of judge fairness.

| Bias Detector | Measurement |
|---------------|-------------|
| Position Bias | Flip Rate |
| Verbosity Bias | Bias Rate |
| Self-Enhancement | Model Family Risk |

# Why Multiple Bias Detectors?

No single detector can fully characterize judge behaviour.

For example,

A judge may:

- Have no Position Bias
- But exhibit strong Verbosity Bias

or

- Resist Sycophancy
- Yet produce heavily clustered scores

Combining multiple detectors provides a much more complete understanding of evaluation quality.

---

# Limitations

The implemented detectors provide practical estimates rather than absolute proof of bias.

Some limitations include:

- Small evaluation datasets may reduce statistical significance.
- LLM outputs remain probabilistic.
- Some subtle biases require larger benchmark datasets.
- Additional detectors may be required for specialized domains.

Despite these limitations, the implemented bias analysis significantly improves the reliability and transparency of automated LLM evaluation.

This enables users to assess not only **which response is better**, but also **whether the evaluation process itself is fair, reliable, and trustworthy**.



# Judge Validation

A good evaluation framework should not only evaluate model responses but also verify that the evaluation process itself is reliable.

Large Language Models are probabilistic systems. Even when given the same input multiple times, they may occasionally produce different outputs. If an LLM judge frequently changes its decision for identical responses, its evaluations become difficult to trust.

To measure judge reliability, this framework performs **Test-Retest Validation**.

---

# Test-Retest Reliability

Test-Retest Reliability measures whether the judge reaches the same decision when evaluating identical inputs multiple times.

For every evaluation, the framework repeats the judging process multiple times using the same:

- Question
- Expected Answer
- Answer A
- Answer B

Example:

```
Run 1

Winner → A
```

```
Run 2

Winner → A
```

```
Run 3

Winner → A
```

Since every run produced the same winner, the judge is considered highly consistent.

---

# Why Test-Retest?

Consistency is one of the most important characteristics of any evaluation system.

An unreliable judge may produce:

```
Run 1 → A

Run 2 → B

Run 3 → A
```

Such inconsistency reduces confidence in automated evaluation.

By repeatedly evaluating identical inputs, the framework estimates the stability of the judge.

---

# Consistency Rate

The framework calculates a consistency percentage.

Formula

```
                     Consistent Runs
Consistency = ------------------------------ × 100
                     Total Validation Runs
```

Example

```
Runs

A

A

A
```

Consistency

```
100%
```

Example

```
Runs

A

B

A
```

Consistency

```
66.67%
```

Higher values indicate greater judge reliability.

---

# Validation Output

Each validation result contains:

- Winners across all runs
- Number of consistent runs
- Total validation runs
- Consistency percentage
- Stability status

Example

```json
{

        "A"

}
```

# Validation Report

This provides a high-level overview of judge reliability across the entire dataset.

---

# Why Judge Validation Matters

Without validation, users must assume that the judge behaves consistently.

By explicitly measuring consistency, the framework provides evidence that the evaluation process is reliable rather than relying on assumptions.

Judge validation therefore complements bias detection by measuring stability instead of fairness.

# Limitations

Although the framework provides a comprehensive evaluation pipeline, several limitations remain.

## Local Model Performance

Evaluation quality depends on the selected local LLMs.

Using stronger models generally produces more reliable evaluations.

---

## Small Datasets

Bias measurements become more reliable with larger datasets.

Very small test suites may not produce statistically significant results.

---

## LLM Variability

Large Language Models are probabilistic systems.

Although judge validation measures consistency, slight variations may still occur across repeated evaluations.

---

## Computational Cost

Running multiple evaluations, bias detectors, and validation experiments increases execution time.

This trade-off was accepted in favor of more comprehensive evaluation.

---

## Limited Bias Coverage

The framework currently detects five common evaluation biases.

Additional biases may exist and could be incorporated in future versions.

---

# Conclusion

This project presents a complete offline LLM evaluation framework designed to evaluate not only the quality of generated responses but also the reliability and fairness of the evaluation process itself.

Unlike traditional benchmarking systems that focus solely on response quality, this framework introduces independent judging, quantitative metrics, bias detection, judge validation, structured reporting, and an interactive dashboard within a single modular architecture.

Key achievements of the project include:

- A modular evaluation pipeline
- Independent generator and judge models
- Robust structured parsing
- Comprehensive evaluation metrics
- Five bias detection mechanisms
- Judge reliability validation
- JSON and CSV report generation
- Interactive Streamlit dashboard
- Fully offline execution using Ollama

The modular architecture ensures that the framework can be easily extended with new models, metrics, bias detectors, and reporting capabilities.

Overall, the project demonstrates how automated LLM evaluation can be made more transparent, reproducible, and trustworthy through careful engineering and systematic validation.
